1. Runtime and Configuration
1.1 Core paths and defaults
- Flask template folder: APP_DIR/templates
- Flask static folder: APP_DIR/static
- Config file: APP_DIR/mcweb.env
- Default service: minecraft
- Backup script: APP_DIR/scripts/backup.sh
- Backup directory default: platform default backup dir (typically /home/.../backups)
- Minecraft root default: platform default minecraft root (typically /opt/Minecraft)
- WORLD_DIR default: MINECRAFT_ROOT_DIR/config
- Crash reports dir: MINECRAFT_ROOT_DIR/crash-reports
- Minecraft logs dir: MINECRAFT_ROOT_DIR/logs
- Structured runtime state DB: APP_DIR/data/app_state.sqlite3

1.2 Security and setup
- Setup mode gates app routes until setup is completed.
- CSRF token stored in Flask session (`csrf_token`), accepted from header `X-CSRF-Token` or form field `csrf_token`.
- CSRF enforced on mutating methods except heartbeat endpoints.
- Password checks use `check_password_hash` against configured admin hash.

1.3 Timezone and environment
- DISPLAY_TZ defaults to `Asia/Manila`.
- Process timezone (`TZ`) is forced to DISPLAY_TZ; `time.tzset()` called when available.

2. Control Plane Operation Model (SQLite)
2.1 Durable operation state
- Async mutating actions write operation records in SQLite table `operations`.
- Stored fields include:
  - `op_id`
  - `op_type` (start|stop|backup|restore)
  - `target`
  - `idempotency_key`
  - `status` (intent|in_progress|observed|failed)
  - `checkpoint`
  - `attempt`
  - `intent_at`, `started_at`, `finished_at`
  - `error_code`, `message`
  - `data_json` payload

2.2 Operation API behavior
- `GET /operation-status/<op_id>` returns current operation status payload.
- All control actions `/start`, `/stop`, `/backup`, `/restore-backup` return `202 Accepted` with `op_id` when accepted.

3. Control Routes
3.1 Start (`POST /start`)
- No password required.
- Blocks on low storage.
- Creates durable operation (`start`) with `intent`.
- Worker transitions operation:
  - `in_progress` (checkpoint `worker_started`)
  - `failed` on start command/session write/thread failure
  - `observed` on successful start + session write
- Session start timestamp is written only after successful start command result.
- Supports idempotency key (`X-Idempotency-Key` or `idempotency_key` form):
  - Existing active/observed op returns same `op_id`.
  - Failed op can be resumed (`attempt` increment, `checkpoint=resume_requested`).

3.2 Stop (`POST /stop`)
- Password required.
- Creates durable operation (`stop`) with async worker.
- Worker performs graceful stop, clears session, resets backup schedule, emits cleanup event.
- Operation transitions to `failed` or `observed`.
- Supports idempotency/retry semantics like start.

3.3 Backup (`POST /backup`)
- No password required.
- Now async via operation record (`backup`) and worker thread.
- Worker runs backup script trigger `manual`.
- Operation transitions:
  - `in_progress` -> `observed` on success
  - `failed` on backup failure/thread start failure
- Supports idempotency/resume semantics like start/stop.

3.4 Restore (`POST /restore-backup`)
- Password required.
- Requires `filename`.
- Creates durable `restore` operation and background restore job.
- Operation tracks restore job id in payload and terminal result.
- Supports idempotency:
  - Same key + same target returns existing op.
  - Same key + different target returns `409 idempotency_key_conflict`.
  - Failed op can be resumed (attempt increment + resume checkpoint).

4. Restore Workflow Safety
4.1 Safe extraction
- Zip extraction path traversal is blocked by `_safe_extract_zip` before extraction.
- Unsafe archive paths fail restore with explicit error.

4.2 Failure and restart handling
- If restore stops an active service and restore later fails, workflow attempts controlled restart.
- On success, service is restarted only if it was active before restore.
- Pre-restore snapshot is created before world switch.
- Current world is archived before switching to restored world.

5. Observed State and Runtime Truth
5.1 Observed snapshot endpoint
- `GET /observed-state` returns runtime-observed state from service/filesystem probes plus latest operation summaries.

5.2 Dashboard status derivation
- Dashboard metrics include `observed_state`.
- Runtime display and nav attention derive from observed status, with active operation overlays (starting/shutting) from latest ops.

6. Reconciliation Loop
6.1 Background reconciler
- Background operation reconciler runs periodically.
- It checks active operations (`intent`, `in_progress`) against runtime status.
- It advances stale/timed out operations to `failed` and resolves completed ones to `observed`.
- Handles start/stop/restore/backup op timeout/staleness paths.

6.2 Consistency checks in loop
- Reconciler also runs invariant consistency checks (auto-repair mode) and logs warnings when issues are detected.

7. Consistency and Invariants
7.1 Consistency report builder
- Runtime consistency report validates key invariants:
  - Active service implies session start timestamp exists.
  - Off service implies session start timestamp is cleared.
- Optional auto repair performs safe corrective writes.

7.2 Consistency endpoint
- `GET /consistency-check`
  - Returns report without mutation by default.
  - `auto_repair=1` requires password (`sudo_password`) and executes repair attempts.

8. UI Behavior for Async Operations
8.1 Home page
- Start/stop/backup submissions receive async acceptance.
- Home JS polls `/operation-status/<op_id>` for accepted operations until terminal state.
- UI refreshes metrics on completion and surfaces terminal errors via modal.

8.2 Backups page
- Restore UI still uses restore status stream for progress and now also polls operation status for terminal op failure/observed state.

9. File and Backup Listing
- Backups page includes both zip backups and snapshot directories.
- Snapshot items expose:
  - restore name format `snapshot::<dir>`
  - download URL `/download/backups-snapshot/<dir>`
  - download filename `<dir>.zip`
- Snapshot download route zips snapshot dir before sending.

11. Known Remaining Gaps vs strict intent
- Restore and backup use checkpoint labels, but restore resume is still coarse-grained (worker rerun) rather than strict step-by-step checkpoint continuation for every internal sub-step.
- UI does not render a dedicated generic operation progress panel for all actions; it polls and resolves terminal state for home actions and restore flow.

12. Tests and Verification Status in Repo
- Route coverage tests include async op acceptance and operation status checks.
- Idempotency tests cover start/stop/backup/restore reuse/resume and restore key conflict.
- Reconciler tests cover operation advancement and timeout paths.
- Consistency tests cover invariants and `/consistency-check` password-gated auto-repair behavior.
