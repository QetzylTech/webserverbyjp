1. Live Metrics

Server side: Data to be broadcasted to all conencted flask app clients in json format via sse. 
- Broadcast to all clients: Ram usage, cpu usage per core, cpu frequency, storage utilization, minecraft server status, rcon players online, tick time, auto stop timer, backup status, last backup, next backup, backups folder, live server console logs, live backup activity logs, live control panel activity (actions) log, live control panel logs (errors/system) logs, backup files count, stale worlds count, cleanup last run, cleanup rule version, schedule version, rule last changed by, rules and schedule next run date and time, rules and schedules missed run count.
- Since everything is broadcast to clients, there should be no individual polling from the clients.
Server side: Cadence by state
- When minecraft server is off and there are 0 flask app clients connected, all metrics and logs are paused. Server fetches and broadcasts nothing.
- When minecraft server is on and there are 0 flask app clients connected, all metrics are paused, logs are read every 15s and cached until a client connects so the client receives something instantly, storage is computed every 15s.
- When there are 1 or more flask app clients connected, regardless of minecraft server status, all data are broadcast by the server at 1hz to all clients. Live logs are broadcast as they come (independent of the 1hz cadence).

Server side: Caching and storage refresh
- File lists and counts are cached so that the clients immediately receives something upon connecting, then gets updated by regular server updates. 
- Storage related data like file count, file lists, and cleanup rules display are read once at boot and updated once every 15 seconds or when clients connect or an action or event triggers a change. It is paused if server is off and 0 flask app clients connected.

Server side: Log behavior
- Live Minecraft server logs are paused when server is off and 0 flask app clients are connected. Data is cached so the first client gets something immediately.
- Live logs update normally when server is on
- Live backup logs update in realtime when backup status is queued or running.
- Control panel logs only update when a client is connected or server is on.
- Only te latest 500 lines of non rcon noise log is kept in cache and sent to the client.

Client side: UI behavior
- Update the UI as fast as possible to display data as soon as it is received. Alternative is if this is just an event listener waiting for data to arrive instead of an active refresh rate.
- CSS and JS are cached. Clients ave the app installed as a webservice so when clients open the pages but there is no connection to the server, the ui still load but doesn't load data and just says no connection. Page is repopulated as soon as connection is established.
- If the client opens the app in multiple tabs, just one of those tabs gets the data and it in turn just forwards the data to the other tabs. Basically I want multitabs to still act like a single tab.
- Even when there is no connection to the server, accessing the url should still show the ui and allow page navigation, just without the data and with a banner saying that there is no connection. As soon as server is available, rehydrate the page.
- Entire Client is a shell. All metrics and processess are shell owned. Pages own nothing.
2. Server Runs and Status

Start and status rules
- Only trigger is pressing the start button. No password required.
- Pressing start triggers status queued.
- Starting status shows up when start is pressed but logs have not yet returned the started-up message, Dedicated server took x seconds to load.
- Server status is running when the started-up message is seen in the logs.
- If the flask app boots while the server is already running, status is running.
- Pressing stop and confirming password shows shutting down status.
- Status becomes off when the server is not running.
- Pressing the start button disables it for 10 seconds (10 second cooldown). Regardless of cooldown, pressing the start button twice should not send duplicate start commands.

Stop triggers
- Stop is pressed and password entered.
- Server is running but has 0 players for 3 minutes.
- Storage capacity left is 10% remaining  or when the estimated storage required to complete the backup exceeds available storage (Emergency Shutdown).

Button enablement
- Start button is only clickable when server is off.
- Stop button is only clickable when server is not off.
- Backups button is not clickable when status is starting or shutting down.
- Start and backup buttons are not clickable when remaining storage capacity is 10%  or when the estimated storage required to complete the backup exceeds available storage

3. Backup and Restore
Backups
- Backups have 4 triggers: end of every session when the server is shutdown, backup button is pressed, backup interval is reached, backup restoration is initiated.
- Running a backup script causes the script to write to a file true to indicate that it is running and then write false to indicate otherwise. The flask app accesses this file to update te backup status.
- All backups except auto backups are done by zipping the files and and putting them in the backups directory.
- Auto backups are done through snapshots and only the latest 3 snapshots are kept. Snapshots are also kept in the backups directory.
- Backup filenames reflect purpose: session end, manual, emergency, auto. Backup filenames also have a timestamp.
- All backups can be downloaded but require a password.
- Backup cannot run when a backup is in progress. Prevent simultaneous backup triggers.
- Backup, wen triggered, saves the world, sturns off auto saving, zips or snapshots the world, and then reenable world saving. 
- Backup alerts the server players through rcon about backup starting,  ending successfully, or failed due to a reason.
- Backup will not run when storage is below 10% or when the estimated storage required to complete the backup exceeds available storage

Restore flow
- List of all backups and snapshots in the backups directory are shown. Restore buttons are present beside the download buttons.
- Clicking restore opens the restore pane.
- Once initiated, the restore creates a pre-restore backup zip of the current world (recorded in a database).
- Old world files are moved to the old worlds folder, renamed, and recorded to a database.
- The backup being restored is extracted or copied to the root directory where the old world once was.
- The copied world folder is renamed to a new world and recorded to a database.
- server.properties is updated to reflect the new world folder.
- There is no dedicated undo. To roll back, manually restore using the pre-restore backup via the normal restore flow.
- A live log stream for the restore should be visible in the restore pane.
- Old worlds folder can become stale over time and may need cleanup.
- Restore is disabled when minecraft server is running or when there is an ongoing backup running.
- When Restoring from an invalid or corrupted backup, all changes for that restore reqest is reverted.
- Restore will not run when storage is below 10%

4. Cleanup
Eventually, old backups and stale worlds should be deleted. These are the cleanup rules:
- Separate rules for backups and stale worlds.
- Backup  Rules:
    - Age Rule: Minimum age to start deleting (Days) <int set by user>; hard guard: must be older than 3 days
    - Space Rule: Max disk usage before deleting starts (% used) <int set by user>
    - Count Rule: Newest Session backups to keep  <int count set by user>; hard guard:  minimum of 3 and hard guard set to prevent deleting last  of any backup type.
    - Newest Manual backups to keep  <int count set by user>; hard guard:  minimum of 3 and hard guard set to prevent deleting last  of any backup type.
    - Newest Prerestore backups to keep  <int count set by user>; hard guard:  minimum of 3 and hard guard set to prevent deleting last  of any backup type.
    - Newest Emergency backups to keep  <int count set by user>; hard guard:  minimum of 3 and hard guard set to prevent deleting last  of any backup type.
    - Time Based Rule: Enable or Disabled; Time of cleanup <time set by user>; Repeat <choice: daily, weekly on day, monthly on day>
- Stale World Rules:
    - Age Rule: Minimum age to start deleting (Days) <int set by user>; hard guard: must be older than 3 days
    - Space Rule: Max disk usage before deleting starts (% used) <int set by user>
    - Count Rule: Newest Session backups to keep  <int count set by user>; hard guard:  minimum of 3 and hard guard set to prevent deleting last  of any backup type.
    - Time Based Rule: Enable or Disabled; Time of cleanup <time set by user>; Repeat <choice: daily, weekly on day, monthly on day>
- Cleanup triggers
    - Trigger a cleanup if scheduled cleanup is enabled in the rules.
    - Trigger a cleanup if manual trigger is pressed in the rule pane.
    - Trigger a cleanup if manual cleanup is pressed after selecting files and entering password
    - All manual cleanup triggers are dry run  by default, no  password. Wet runs, actual delete require password and acknowledgement of the dangers.

14. Conflict resolution
    - Only one of {backup, restore, cleanup} can run at a time
    - Some rules:
        - A running server will block backup restoration
        - Backups can only run when server is off or running
        - A backup job will be prioritized over cleanup and restore jobs (backup > restore > cleanup)
    - Simultaneous jobs are queued.
    - Multiple clients initiate the same job:
        - Backups: queue jobs
        - Cleanup: reject subsequent jobs until current is finished
        - Restore: reject subsequent jobs until current is finished
        - Server start: give a cooldown from last trigger and ignore subsequent start triggers until server status returns to off
        - Server Stop: Queue jobs. There is no harm in repeated stop calls. For ui consistency, just disable the stop button when the server status is not running
        - 
    - If jobs conflict (e.g restoring a file that is queued for deletion or deleting a file queued for restoration) the restoration taked precedence.

16. Implemented Behavior Not Explicitly Listed Above
    - Server-side client registry with heartbeats to track connected clients and detect abrupt disconnects.
    - Log streaming tracks file offsets and buffers lines when idle, then drains buffered lines on client connect.
    - Restore log stream is exposed via dedicated SSE channel /stream/restore_logs (separate from main logs).
    - Cleanup scheduler persists last tick per scope and records missed-run metadata in a non-normal state file.
    - Cleanup run history is persisted and attached to metrics/history views.
    - Cleanup blast-radius caps are enforced per run (max percent, absolute max, and minimum per run).
    - Server-side priority/guard enforcement for cleanup actions (reject when backup/restore running or queued).
    - Storage guard calculates free space percent and estimated job size to block start/backup/restore and trigger emergency shutdown.
